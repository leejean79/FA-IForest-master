# EXP3 论证骨架 — 系统处理延迟(集群数据定稿:延迟部分)

> 用途:论文 EXP3 延迟小节脚手架。与吞吐小节(EXP3_throughput_skeleton.md)配套,
>      共同刻画系统"高吞吐低延迟"基石。
> 数据:集群延迟 run,donors × P∈{1,2,4,6} × 3,detector=IKS,BACKLOG,
>      JOB_LOAD_RATE=2000(经探测不堆积,见 §2.2),source 单分区,detectionParallelism=P。
> 延迟口径:raw_lat = scoreTime − ingestionTime(数据进入系统→打分完成,不含磁盘 dump)。
> 数据范围:延迟仅用 donors。http 因灌入时 record timestamp 大量缺失(ingestionTime 91.7%
>          为 0,已知数据采集限制),不纳入延迟分析,仅用于吞吐。

---

## 1. 论点(claim)

**在保证不堆积(producer rate < 系统处理能力)的实时摄入下,系统单条数据的处理延迟
(摄入→打分完成)为数十毫秒量级(donors 37–53ms 中位)。延迟随并行度轻微上升
(P=1 的 37ms → P=6 的 53ms),而非下降——与吞吐结论一致:在 source 单分区瓶颈下,
增加并行度不带来吞吐提升,反因跨 subtask 传输/序列化开销使单条延迟略增。**

这与吞吐部分构成统一的系统级叙事:单分区保时序的约束下,并行度增加只增开销。

## 2. 机制/背景

### 2.1 延迟口径(实时摄入,不堆积)
- raw_lat = scoreTime − ingestionTime。ingestionTime = Kafka record timestamp(摄入时刻),
  scoreTime = 打分完成时刻。截止点为打分完成,不含 Step 7 一次性磁盘 dump(实验观测手段)。
- 须在"不堆积"下测:producer rate < 系统处理能力时,数据来即处理,raw_lat = 纯处理延迟;
  若堆积(rate 过高),raw_lat 含 backlog 排队、失真。

### 2.2 rate 选值与不堆积验证
- 选 rate=2000(远低于 source 单分区上限 ~7000-9600,见吞吐部分),确保不堆积。
- 不堆积验证(每 run):按摄入序比较前/后半段 raw_lat,比值 ≤2 即不堆积。
  donors 全部 run backlog_ratio 0.91–0.98(前后半段一致)→ 确认不堆积,raw_lat = 真实处理延迟。

### 2.3 数据质量过滤
- 过滤脏 ingestionTime(0/缺失,donors 占 0.9%):raw_lat 取中位,对少数脏值不敏感;
  过滤后基于 99.1% 干净记录,结论可靠。

## 3. 证据(集群 donors,P × 3 repeats)

### 3.1 处理延迟 × 并行度(raw_lat 中位,ms)

| P | raw_lat median(均值) | 三次 repeat | p95(约) |
|---|---|---|---|
| 1 | 37.0 | 36 / 39 / 36 | ~85 |
| 2 | 46.7 | 44 / 51 / 45 | ~95 |
| 4 | 50.3 | 51 / 50 / 50 | ~97 |
| 6 | 53.3 | 52 / 54 / 54 | ~98 |

(吞吐均 ≈2010 rps,确认各 P 均不堆积、负载一致;backlog_ratio 0.91–0.98。)

### 3.2 关键观察

**(A) 处理延迟为数十毫秒量级。** donors 中位 37–53ms,p95 ~85–98ms。证明系统在实时摄入下
具备低延迟处理能力(摄入→打分完成于数十毫秒)。

**(B) 延迟随并行度轻微上升(37→53ms),非下降。** 三次 repeat 一致(波动 ≤5ms),趋势稳健。
→ 解释:source 单分区限制端到端吞吐(~7-9.6k rps),增加并行度并无更多数据可分担,
反引入跨 subtask 网络传输 + 序列化开销,单条延迟略增。

**(C) 与吞吐结论统一。** 吞吐部分:并行度增加不提升端到端吞吐(被 source 锁定),P=6 略降。
延迟部分:并行度增加使延迟略升。二者同源——单分区瓶颈下,并行度只增协调开销,
不改善吞吐也不降低延迟。这是本系统"时序优先"架构的一致代价。

**(D) http 干净子集佐证趋势(稳健性)。** http 虽因 ingestionTime 缺失 + 处理吞吐较低
(~1660 rps < donors ~2010,致 rate=2000 下 P=6 堆积)不作延迟主数据,但其不堆积的
P∈{1,2,4} 子集(raw_lat 中位 44→53→72ms,backlog_ratio≈1)呈现与 donors 一致的
"延迟随并行度上升"趋势,且上升更陡。→ 延迟随并行度略升不是 donors 单数据集孤证,
有 http 佐证,结论稳健。

## 4. 结论与权衡叙事(与吞吐合写)

系统级"高吞吐低延迟"的诚实刻画:
- **低延迟达成**:实时摄入(不堆积)下,处理延迟数十毫秒(37–53ms),满足流式异常检测的实时性。
- **延迟/吞吐均不随并行度改善**:source 单分区(保检测时序)是统一瓶颈——它既限制端到端吞吐
  (~7-9.6k rps 不扩展),也使延迟随并行度略升(协调开销)。
- **核心权衡**:时序保证(单分区)↔ 并行扩展(吞吐/延迟)。本系统选择时序正确优先,
  代价是端到端吞吐与延迟不随并行度改善。多分区+算子内重排(方向 B)为 future work。

## 5. 写作风险与防线
- 不夸大"分布式降低延迟"——数据是延迟随并行度略升,如实写,并解释为开销主导(与吞吐自洽)。
- 强调延迟低(数十 ms)是在"不堆积/实时摄入"前提下;须说明 rate 选值与不堆积验证(§2.2)。
- 延迟仅 donors:诚实声明 http 因 ingestionTime 大量缺失(数据采集限制)不纳入延迟,
  不掩盖、作为已知 limitation。单数据集(donors,大流、10 维)足以支撑延迟扩展性结论。
- p95 与 median 一并报(median 37-53、p95 85-98),避免只报乐观值。

## 6. 数据完整性说明(须写入)
- ingestionTime 脏值(0/缺失):donors 0.9%(过滤,不影响中位);http 91.7%(故 http 不用于延迟)。
- stable-K:延迟 run 已放宽容忍(若已应用),scores 完整;raw_lat 取全程中位,不堆积已验证。
